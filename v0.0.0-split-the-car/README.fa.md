## مقدمه

در اولین مرحله از ساخت این پروژه، هدف ما آشنایی با یکی از پایه‌ای‌ترین مفاهیم معماری ECS یعنی **Entity** و **Component** است.

برای درک بهتر این مفاهیم، ابتدا باید تفاوت دیدگاه بین برنامه‌نویسی شی‌گرا (Object-Oriented Programming) و طراحی داده‌محور (Data-Oriented Design) را بررسی کنیم.

در برنامه‌نویسی شی‌گرا، معمولاً هر مفهوم از دنیای واقعی را به شکل یک **Object** مدل می‌کنیم. این Object توسط یک **Class** تعریف می‌شود و تمام اطلاعات مربوط به آن در همان کلاس قرار می‌گیرد.

برای مثال، اگر بخواهیم یک ماشین را مدل کنیم، می‌توانیم کلاسی مانند `Car` بسازیم و اطلاعاتی مانند رنگ، مدل و مالک ماشین را به عنوان داده‌های آن کلاس ذخیره کنیم.

در این روش، هر Object تمام اطلاعات مربوط به خودش را در یک مکان نگهداری می‌کند و مدیریت آن از طریق همان Object انجام می‌شود.

## تفاوت دیدگاه DOD

در طراحی داده‌محور (Data-Oriented Design)، تمرکز اصلی روی داده‌ها و نحوه سازمان‌دهی و پردازش آن‌ها است.

در این روش، به جای اینکه تمام اطلاعات یک موجودیت را در یک Object بزرگ قرار دهیم، داده‌ها را به بخش‌های کوچک‌تر تقسیم می‌کنیم. هر بخش فقط مسئول نگهداری یک نوع داده مشخص است.

برای مثال، اطلاعات یک ماشین در روش شی‌گرا ممکن است در یک Object به نام `Car` قرار داشته باشد. اما در روش داده‌محور، این اطلاعات به Componentهای مختلف تقسیم می‌شوند. رنگ، مدل، مالک و سایر ویژگی‌ها هرکدام Component جداگانه‌ای خواهند داشت.

در ECS، این تقسیم‌بندی باعث می‌شود که یک موجودیت از دو بخش اصلی تشکیل شود:

- **Entity** که هویت و شناسه موجودیت را مشخص می‌کند.
- **Component** که داده‌ها و ویژگی‌های مربوط به آن موجودیت را نگهداری می‌کند.

بنابراین یک ماشین دیگر یک Object مستقل با تمام اطلاعات داخلی خود نیست، بلکه یک Entity است که Componentهای مختلفی به آن متصل شده‌اند.

## Entity چیست؟

در ECS، Entity معمولاً یک شناسه ساده است که برای تشخیص موجودیت‌ها استفاده می‌شود.

برخلاف Object در برنامه‌نویسی شی‌گرا، Entity خودش اطلاعاتی مانند رنگ، مدل یا سایر ویژگی‌ها را نگهداری نمی‌کند.

هدف Entity فقط این است که مشخص کند داده‌های مختلف متعلق به کدام موجودیت هستند.

برای مثال، یک Entity می‌تواند نماینده یک ماشین باشد و Componentهای مربوط به آن ماشین از طریق همان شناسه به آن متصل شوند.

به بیان ساده:

Entity مشخص می‌کند «این داده‌ها متعلق به چه کسی هستند» و Component مشخص می‌کند «چه داده‌ای وجود دارد».

## Component چیست؟

Componentها بخش‌هایی از داده هستند که از Object اصلی جدا شده‌اند.

هر Component فقط یک نوع خاص از اطلاعات را نگهداری می‌کند. برای مثال، می‌توانیم یک Component برای رنگ، یک Component برای مدل و یک Component برای مالک داشته باشیم.

در ECS، به جای اینکه تمام ویژگی‌های یک ماشین را در یک ساختار بزرگ ذخیره کنیم، آن‌ها را به بخش‌های کوچک‌تر تقسیم می‌کنیم.

این کار باعث می‌شود داده‌ها مستقل‌تر باشند و در مراحل بعدی بتوان آن‌ها را راحت‌تر مدیریت و پردازش کرد.

## تبدیل Object به Entity و Component

در برنامه‌نویسی شی‌گرا، یک ماشین معمولاً به شکل یک Object مدل می‌شود که تمام ویژگی‌های آن درون همان Object قرار دارد.

اما در ECS، همان ماشین به یک Entity تبدیل می‌شود و ویژگی‌های آن به Componentهای جداگانه منتقل می‌شوند.

در نتیجه، Entity فقط نقش شناسایی را دارد و Componentها اطلاعات واقعی موجودیت را نگهداری می‌کنند.

این همان ایده اصلی ECS است: جدا کردن هویت یک موجودیت از داده‌های آن.

## مقایسه با کد

برای درک بهتر تفاوت بین این دو روش، در این نسخه دو نمونه کد قرار داده شده است که یک مفهوم یکسان را یک بار با روش شی‌گرا و یک بار با روش داده‌محور پیاده‌سازی می‌کنند. پیشنهاد می‌شود برای درک کامل‌تر، هر دو کد را مطالعه کنید.

## رفتارها و Systemها

در ECS علاوه بر Entity و Component، مفهوم مهم دیگری به نام **System** وجود دارد.

در برنامه‌نویسی شی‌گرا، رفتارهای یک Object معمولاً داخل همان کلاس و به شکل توابع عضو قرار می‌گیرند.

اما در ECS، رفتارها از داده‌ها جدا می‌شوند و Systemها مسئول پردازش Componentها خواهند بود.

در این نسخه هنوز وارد مفهوم System و رفتارها نمی‌شویم، زیرا هدف نسخه 0.0.0 فقط درک Entity و Component و نحوه جداسازی داده‌ها از Object است.

مفهوم System و نحوه مدیریت رفتارها در مراحل بعدی پروژه بررسی و به ECS اضافه خواهد شد.

## بیلد و اجرا

برای ساخت پروژه، ابتدا وارد پوشه `v0.0.0-split-the-car` شوید و سپس دستورات زیر را در ترمینال اجرا کنید:

```bash
mkdir build
cd build
cmake ..
cmake --build .
```

پس از اتمام فرایند بیلد، فایل‌های اجرایی `cars` و `splited_cars` (یا در ویندوز `cars.exe` و `splited_cars.exe`) در پوشه خروجی ساخته می‌شوند. بسته به کامپایلر و ژنراتور مورد استفاده، ممکن است فایل‌های اجرایی در زیرپوشه‌هایی مانند `Debug` یا `Release` قرار گیرند.

برای اجرای نمونه‌ها، از داخل پوشه حاوی فایل اجرایی، دستور مناسب را اجرا کنید:

```bash
./cars
./splited_cars
```

در ویندوز و PowerShell:

```powershell
.\cars.exe
.\splited_cars.exe
```
